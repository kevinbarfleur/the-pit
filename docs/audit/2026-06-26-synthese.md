# Synthese de l'audit - 2026-06-26

## Verdict court

Le projet n'est pas un plat de spaghetti. Il a deja des fondations rares pour un
prototype LÖVE/Lua : simulation deterministe, rendu separe, bus d'evenements,
tests headless larges, exports visuels, design system in-engine, SFX/music
centralises, snapshot async, golden tests.

Le vrai risque n'est pas "le code est mauvais". Le risque est plus insidieux :
plusieurs verites peuvent vite cohabiter si les docs ne sont pas tenues courtes.
Un nettoyage documentaire a ete fait le 2026-06-26: `CLAUDE.md`, les briefs
principaux et les docs actives marquent maintenant les anciennes decisions comme
historiques. Il faut garder cette discipline pendant les prochaines passes de
simulation, UI et economie.

Si rien n'est fait, la dette va surtout exploser dans les zones de haute
frequence : `src/scenes/build.lua`, le storybook/design system, les generateurs
visuels, les constantes d'equilibrage, et les transitions entre scenes.

## Ce qui est solide

1. **Determinisme et firewall SIM/RENDER**
   `tools/check.sh` passe. Les gardes interdisent RNG global et rendu dans la
   simulation. Les modules critiques utilisent surtout des tableaux ordonnes et
   `ipairs`, coherents avec Lua 5.1 et la contrainte de determinisme.

2. **Boucle moteur saine**
   `main.lua` possede un `love.run` a timestep fixe 1/60, avec accumulator et
   plafond de rattrapage. C'est conforme a la logique fixed timestep : simuler
   avec un pas constant, rendre separement, eviter la spirale de rattrapage.

3. **Architecture de combat en bonne direction**
   `src/combat/arena.lua` est maintenant SIM pure, et `src/render/arena_draw.lua`
   consomme les evenements pour jouer animations, nombres, sons et shake. C'est
   exactement la bonne separation pour un autobattler async.

4. **Design system concret**
   `src/ui/theme.lua`, `src/ui/draw.lua`, `src/ui/button.lua`, `src/ui/panel.lua`,
   `src/ui/slot.lua`, `src/ui/tooltip.lua`, `src/scenes/designsystem.lua` donnent
   une base de production. Ce n'est pas juste une intention dans un doc.

5. **Pipeline de revue visuelle deja present**
   `love . --shoot=all --shoot-size=1280x720` genere des captures exploitables.
   C'est un levier majeur pour garder la coherence d'un jeu visuel.

## Les risques principaux

1. **Dérive de source de vérité**
   Le nettoyage a reduit le risque, mais il reste permanent: chaque nouvelle
   feature doit mettre a jour `CLAUDE.md` ou le handoff actif si elle change une
   decision de gameplay, et les vieilles references techniques ne doivent pas
   redevenir des specs.

2. **`build.lua` est le plus gros point de fragilite**
   3532 lignes, beaucoup de responsabilites : layout, input, boutique, board,
   bench, commandant, drag, fusion, tooltips, auras, effets visuels, SFX. Il est
   encore comprehensible, mais il ne faut pas attendre plus.

3. **Politique de chrome visuel absente**
   Les captures montrent une palette coherente, mais pas encore une grammaire de
   scene stable. Menu, build, combat, resume, grimoire, relicpick et storybook
   ont chacun leur intensite theatrale. Il faut definir quelles familles
   d'ecrans ont quel cadre, quelle densite, quels ornements, quels boutons.

4. **Scaling rendu a stabiliser**
   Le code actuel remplit la fenetre avec `Viewport.scale` fractionnaire et des
   safe areas. Les anciens textes integer-only ont ete corriges dans les briefs
   principaux; il faut maintenant proteger ce choix par captures et tests.

5. **Transition de scene trop brute**
   `host.goto` reconstruit/switch directement. Le son est plus mature que la
   transition visuelle : la musique crossfade par morceau, mais il n'y a pas
   encore de service de transition ecran generique.

6. **Tuning encore trop enfoui dans le code**
   `src/run/state.lua` et `src/combat/arena.lua` contiennent beaucoup de
   constantes et d'historique de tuning. C'est acceptable en prototype, mais pas
   pour iterer vite sur l'equilibrage.

## Recommandation de priorite

Priorite 0 : continuer a stabiliser les sources de verite. Le gros nettoyage est
fait; le travail restant est de ne pas laisser revenir des handoffs longs et
contradictoires.

Priorite 1 : extraire `build.lua` par zones pures et composants UI, sans
refactor abstrait massif. Le but n'est pas de "faire propre" pour le plaisir,
mais de retirer le risque du fichier qui change le plus.

Priorite 2 : ecrire une politique UI/DA appliquee par screenshots : chrome par
famille d'ecran, niveau d'ornement, typographie, densite, CTA, overlays.

Priorite 3 : ajouter un petit service de transition autour de `host.goto`.
Il doit rendre le changement de scene perceptible sans toucher la simulation.

Priorite 4 : extraire les tables de tuning economie/combat dans des modules de
donnees et brancher les outils de simulation/balance dessus.

## Formule directrice

Le projet doit rester construit autour de trois frontieres :

- **SIM** : deterministe, seedee, ordonnee, testee, aucune dependance rendu/audio/temps mur.
- **PRESENTATION** : effets, sons, shake, transitions, animations, lisibilite, zero impact sim.
- **DATA/TUNING** : chiffres et contenus faciles a lire, comparer, simuler et exporter.

Tout ce qui traverse ces frontieres doit passer par des contrats explicites :
event bus, snapshot, comp baked, module UI, ou table de tuning.
