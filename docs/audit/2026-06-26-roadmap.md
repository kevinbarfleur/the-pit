# Roadmap d'amelioration - audit 2026-06-26

Cette roadmap vise la consolidation, pas la refonte totale. Le projet a deja de
bonnes fondations ; il faut maintenant reduire les zones ou l'intention se perd.

## Phase 0 - Source de verite et docs actives

Objectif : arreter les contradictions avant qu'elles guident de mauvais choix.

Statut apres nettoyage documentaire du 2026-06-26 : l'essentiel est fait.
`CLAUDE.md`, `docs/README.md`, `engine-architecture.md` et les briefs principaux
ont ete remis a jour. Les points ci-dessous deviennent des gardes continus.

Actions :

- Garder `CLAUDE.md` et `docs/README.md` courts et actuels.
- Marquer explicitement tout ancien document conserve comme historique ou
  reference technique limitee.
- Corriger `src/scenes/designsystem.lua` pour ne plus presenter de relique
  `cryptic` comme etat canonique si le modele lisible est definitif.
- Corriger le commentaire High DPI contradictoire entre `conf.lua` et `main.lua`.

Acceptance criteria :

- Une recherche `rg -n "cryptic|hidden|identif|lure|linear"` dans les docs actives
  ne retourne plus de decision ancienne non marquee comme historique/supersedee.
- Les docs actives pointent vers `CLAUDE.md` pour les decisions de gameplay.
- `sh tools/check.sh` passe.

## Phase 1 - `build.lua` sous controle

Objectif : reduire le fichier le plus dangereux sans casser la scene.

Ordre recommande :

1. Extraire `build_layout`.
2. Extraire `build_comp` / `comp_builder`.
3. Extraire rendu shop/board/tooltips en composants UI.
4. Extraire FX de build.

Acceptance criteria :

- `src/scenes/build.lua` descend sous 2200 lignes.
- Les modules extraits ont des tests headless quand ils sont purs.
- `love . --shoot=build --shoot-size=1280x720` reste visuellement equivalent.
- Aucun nouveau module extrait ne depend de `love.graphics` s'il est logique/layout.

## Phase 2 - Politique UI/DA et screenshots

Objectif : regler le probleme de coherence de bout en bout.

Actions :

- Creer une note active de politique de chrome, par exemple
  `docs/audit/chrome-policy.md` ou `docs/research/chrome-policy.md`.
- Classer chaque scene dans une famille : run utilitaire, rituel, codex,
  system/dev.
- Pour chaque famille, definir :
  - niveau de chrome ;
  - type de fond ;
  - usage des grands ornements ;
  - style de CTA ;
  - densite max ;
  - regles de modal/tooltip.
- Ajouter dans `designsystem.lua` les composants vraiment canoniques :
  tabs, modal, tooltip riche, relic card, monster card, shop card, combat unit
  plate, summary row.
- Nettoyer les etats obsoletes du storybook.

Acceptance criteria :

- Les captures `menu`, `build`, `combat`, `summary`, `relicpick`, `grimoire`,
  `system` ont une justification de chrome documentee.
- Aucun ecran de production n'introduit un nouveau style de bouton/panel hors
  design system sans l'ajouter au storybook.
- Les petits defauts visibles sont corriges, par exemple l'espacement du compteur
  `BESTIARY 110/110` dans le grimoire.

## Phase 3 - Transitions et feedback contract

Objectif : rendre les changements de scene et actions majeures coherents.

Actions :

- Ajouter un service `Transition`.
- Decouper `host.goto` en `goto` transitionne et `gotoNow` immediat.
- Creer une table action -> feedback :
  visuel, SFX, delai, payoff, echec.
- Introduire `src/audio/cues.lua`.
- Remplacer progressivement les `SFX.play("...")` directs dans les scenes par
  des cues semantiques.

Acceptance criteria :

- Build -> Combat, Combat -> Summary, Summary -> Build/RelicPick, RelicPick ->
  Build ont une transition perceptible mais courte.
- La transition ne modifie jamais le run state ni les seeds.
- Les actions principales ont un feedback documente et homogene.

## Phase 4 - Tuning data et simulation

Objectif : rendre l'equilibrage plus lisible et moins enfoui.

Actions :

- Extraire `src/data/economy.lua`.
- Extraire `src/combat/tuning.lua`.
- Brancher les outils `lab`, `bands`, `scenarios` sur ces tables.
- Ajouter des rapports de balance lisibles par archetype :
  winrate, duree combat, damage causes, sur-sustain, one-shot prevention,
  impact relique/commandant.

Acceptance criteria :

- Changer une courbe de shop ou une valeur de fatigue ne demande pas d'editer la
  logique de `RunState` ou `Arena`.
- Les rapports de simulation restent deterministes.
- Un golden de meta reference existe pour comparer les changements de tuning.

## Phase 5 - Outils visuels et perf

Objectif : rendre la qualite visible durable.

Actions :

- Ajouter un outil `tools/visual_check.sh` qui lance des captures de base et
  verifie existence/dimensions/pixels non vides.
- Ajouter un mode dev avec stats par scene :
  draw calls, texture memory, canvas switches, allocations Lua approximatives.
- Ajouter une checklist visuelle dans chaque PR/iteration :
  captures avant/apres, ecrans touches, hover/modal, resolution testee.

Acceptance criteria :

- Une regression "ecran vide" ou "capture non generee" est detectee sans oeil
  humain.
- Les captures restent disponibles pour jugement humain, car la coherence DA ne
  se valide pas par hash seul.
- Les scenes build/combat ont une baseline perf simple.

## Ne pas faire maintenant

- Ne pas redecouper `arena.lua` violemment tant que le gameplay continue de
  bouger. Sortir le tuning d'abord.
- Ne pas supprimer `Forge` juste parce qu'il est ancien. Il sert encore de moteur
  de bake et d'iconographie. Il faut limiter sa surface, pas l'arracher.
- Ne pas ajouter un ECS complet. Le projet n'a pas un probleme qui justifie cette
  complexite.
- Ne pas chercher un systeme UI universel abstrait. Le kit actuel suffit ; il lui
  manque surtout des regles et quelques primitives.
- Ne pas rendre les transitions longues. Le jeu est un autobattler a boucle
  courte ; la ceremonie doit servir le rythme.

## Ordre de travail conseille

1. Faire Phase 0 en premier. C'est court et ca evite les mauvais choix.
2. Faire une premiere extraction de `build_layout`, avec tests.
3. Ecrire `chrome-policy.md` et corriger les 2-3 defauts visibles dans les
   captures actuelles.
4. Ajouter `Transition` minimal pour 3 chemins : build->combat, combat->summary,
   relicpick->build.
5. Extraire `economy.lua`, puis seulement apres attaquer le tuning combat.
